# Sprint Planning

You are planning a sprint for Groundwork AI, an AI development and education consultancy.

## When to Use This Skill

Use this skill when preparing for sprint planning, selecting work for a sprint, setting sprint goals, or reviewing sprint capacity.

## Sprint Planning Process

1. **Review previous sprint.** What shipped? What carried over? Why? Learn from the pattern.
2. **Assess capacity.** Account for each team member's availability (PTO, client meetings, workshops). Assume 70% of available hours are productive sprint work — the rest goes to meetings, reviews, and context switching.
3. **Set the sprint goal.** One sentence describing the business outcome this sprint delivers. "Complete API integration for Acme Corp audit" — not "work on various tasks."
4. **Select work items.** Pull from the prioritized backlog. Every item must have: clear description, acceptance criteria, effort estimate, and an assignee. Total committed effort should not exceed 80% of capacity — leave room for the unexpected.
5. **Identify dependencies.** Map which items depend on each other or on external inputs (client feedback, design assets, API access). Schedule dependent work accordingly.
6. **Confirm commitment.** Review the sprint plan with the team. Everyone should understand the goal and agree the workload is achievable.

## Capacity Planning

For a small team, use a simple points or hours system:
- **Small (S):** Half a day or less
- **Medium (M):** 1-2 days
- **Large (L):** 3-5 days
- **X-Large (XL):** Break it down further — XL items should not enter a sprint as-is

## Sprint Ceremonies (Lightweight)

- **Planning (Day 1):** 30-60 minutes. Set goal, select work, confirm assignments.
- **Daily Async Standup:** Text update — done, doing, blocked. No meeting unless blockers need discussion.
- **Mid-Sprint Check:** 15 minutes. On track? Any scope adjustments needed?
- **Demo (Final Day):** 30 minutes. Show what shipped. Celebrate wins.
- **Retro (Biweekly):** 15 minutes. One thing to keep, one thing to change.

## Rules

- Never commit to more work than the team has capacity for. Overcommitting erodes trust and quality.
- Every sprint item must have acceptance criteria before it enters the sprint.
- Carry-over items get priority review — if something carries over twice, it's either too big or not important enough.
- Protect the sprint: new requests go to the backlog unless they're genuinely urgent (blocking a client delivery).
